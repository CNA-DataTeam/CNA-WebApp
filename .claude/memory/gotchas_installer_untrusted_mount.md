**ROOT CAUSE CORRECTION (May 2026, empirically verified via remote control on build 26200):** This is NOT an FSLogix / AppData-redirection / install-*location* problem. It is **Windows RedirectionGuard** (`ProcessRedirectionTrustPolicy` / `EnforceRedirectionTrust`, process-mitigation enum 16, flag 0x1) triggered by **Mark-of-the-Web** on the downloaded installer .exe. Proof: a fresh install to plain `C:\CNA-WebApp` (outside any profile) ALSO fails with 448, and right-click → Properties → **Unblock** on the installer fixes it. The failure is at `uv pip install` → "Failed to inspect Python interpreter from active virtual environment at `.venv\Scripts\python.exe` … The path cannot be **traversed** … untrusted mount point" — i.e. on *traversing* uv's managed-Python junction, not on *creating* it (which is why the old `mklink`-creation probe always false-passes). Verified facts: (1) a process that opts into EnforceRedirectionTrust is blocked from following a non-admin-created junction; (2) the mitigation **is inherited by child processes** (so the MOTW installer's `uv` child inherits it) — therefore self-unblock+relaunch-as-child does NOT work; (3) a **scheduled task** created by a mitigated process does NOT inherit it (spawned by the Task Scheduler service), so re-parenting setup.bat through a one-shot scheduled task is the viable in-installer fix (fallback: tell the user to Unblock + rerun, the only thing proven to work on a real affected, possibly task-creation-locked-down machine). Local repro/validation rig: `simulate-redirection-guard.ps1` at repo root. The FSLogix framing below is the original (superseded) theory.

**uv-VERSION HYPOTHESIS — DISPROVEN (May 2026).** Checked uv on a real affected machine: `uv 0.11.14` (2026-05-12) — NEWER than the dev box's `0.11.6` (2026-04-09) that could NOT reproduce the bug. So updating uv is NOT the fix; the failing machine already has a newer uv. The real differentiator is most likely **the absence of a real system Python on affected machines** (only the WindowsApps stub), which forces uv down the managed-Python+junction path that gets traversed; the dev box has system Python (3.11/3.13) which steers uv away from that path, so it can't faithfully reproduce the uv-level 448 even with managed mode forced. Bottom line: don't chase the exact uv-internal trigger. **The fix is to run uv OUTSIDE the MOTW-mitigated process tree** — proven on real affected machines by BOTH (a) unblocking the installer and (b) running setup.bat manually from a clean shell. Recommended fix: re-parent the setup.bat step via a one-shot scheduled task (scheduler service spawns it → no inherited mitigation), with unblock-and-rerun as the documented fallback. The dev simulator (`simulate-redirection-guard.ps1`) reproduces the kernel mechanism + inheritance + scheduler-escape, but NOT the uv-level failure; use a real affected machine for end-to-end fix validation.

**IMPLEMENTED 2026-05-21 in `CNA-Console-Installer.iss`:** `RunSetupViaScheduledTask` runs setup.bat via a one-shot `schtasks` job (self-contained wrapper + exit-code marker under `C:\ProgramData\CNAConsoleSetup` — a no-space path so /tr needs no quoting), polling the marker (≤20 min). If task creation is blocked it falls back to a direct in-process `Exec`. `ShowSetupFailedDialog` shows a generic, friendly guided "finish setup by hand" dialog (with an **Open Install Folder** button → `explorer.exe`) for ANY artifact-missing failure, only after the layered run. All `CanCreateJunction` / Public-relaunch / `TerminateInstaller` / branched-tip logic removed. Gotcha: on Inno 6.7.1 `CreateCustomForm(W,H,KeepX,KeepY)` needs 4 params (older docs show 0). Verified: compiles clean (ISCC at `%LOCALAPPDATA%\Programs\Inno Setup 6`). On corporate machines where the Windows profile is a mounted container (FSLogix) or AppData is redirected to a network share, `uv venv` fails with `os error 448` ("The path cannot be traversed because it contains an untrusted mount point") at the step **"Failed to create Python minor version link directory"**.

**Root cause:** uv builds the venv by creating a **directory junction** for the Python minor-version link (its transparent patch-upgrade mechanism — e.g. `cpython-3.11` → `cpython-3.11.15-...` under `UV_PYTHON_INSTALL_DIR`). Creating a junction whose path is inside an untrusted mount is what the filesystem blocks with 448. `UV_LINK_MODE=copy` does NOT help — that only governs package installs into site-packages, not the minor-version junction. Pinning `UV_PYTHON_INSTALL_DIR`/`UV_CACHE_DIR` under the install tree doesn't help either if the install tree is itself inside the mount.

**The detection trap (fixed May 2026):** The installer's pre-check originally walked the install path's ancestry calling `GetFileAttributesW` looking for `FILE_ATTRIBUTE_REPARSE_POINT`. An FSLogix profile is a mounted **VHD volume**, not a reparse point — so the attribute check never fires, the pre-check passes, and uv dies halfway through. The post-install validator then catches it (the "Setup finished, but the app is not ready" dialog).

**Why:** you cannot reliably *infer* the untrusted-mount condition from path attributes. You must *perform the operation*.

**How to apply:** the fix in `CNA-Console-Installer.iss` is `CanCreateJunction(Dir)` — it actually runs `mklink /J` in the install dir (no admin / developer mode needed for junctions) and checks the result. If it can't create one, Setup relaunches itself `/SILENT /DIR="C:\Users\Public\CNA-WebApp"` (off-profile, junctions allowed). There's a loop guard: if we're already at the Public path and it STILL fails, show a hard error instead of relaunching forever. See [[project_installer_flow.md]] for the rest of the install sequence.

**A Public install is self-contained — leftover LocalAppData files do NOT cross-contaminate.** Common support question: a user fails to install to `%LOCALAPPDATA%`, then retries at Public *without cleaning up*, and Public also fails — are the LocalAppData leftovers the cause? No. The probe, the clone (`{app}\setup.bat` check), and `setup.bat`'s `UV_PYTHON_INSTALL_DIR`/`UV_CACHE_DIR`/`VENV_DIR` are all pinned to the chosen install dir, so a Public run never reads the old LocalAppData tree. If Public *also* throws 448, it almost certainly means **Public itself can't host junctions** (whole `C:\Users` or disk virtualized, or a machine-wide junction-block policy) → IT. Prove it with a bare `mklink /J "C:\Users\Public\_probe" "C:\Users\Public"` (uses zero CNA files). Cleanup is harmless but won't fix a real Public block.

**Diagnostic discipline for support calls:** two `mklink /J` probes (one in `%LOCALAPPDATA%`, one in `C:\Users\Public`) classify the machine in 10 seconds. Both fail → read the actual error (448 vs. access-denied/policy vs. non-NTFS), try `C:\Temp`, else IT. Both pass but install still fails → it's NOT mount points (most likely AV quarantine of `_internal\python311.dll`, or proxy/SSL blocking the uv/Python/PyPI downloads); re-run `setup.bat` *non-silent* from the install dir to see the real error, since the installer's `{tmp}` logs are deleted on exit. Full guide: `INSTALL-TROUBLESHOOTING.md` at repo root.

**"Manual setup.bat works but the installer fails every time" is usually a skip-ahead illusion.** `setup.bat` is idempotent: it prints "Python 3.11 already installed. Skipping." and "Virtual environment already exists." when a prior partial run left `.venv`/`.uv\python` behind — and those are the exact `uv venv`/`uv python install` steps that throw 448. So a manual re-run can "succeed" only because it skipped the failing operation. Diagnostic question: did manual setup say "Creating virtual environment" or "already exists"? Decisive test: `rmdir /s /q .venv & rmdir /s /q .uv` in that folder, then re-run `setup.bat`. If it now 448s, the location genuinely can't build the venv (bare junction *creation* via the probe ≠ uv *traversing into* the junction to extract Python — a mount can block traversal while allowing creation) → use Public. If it still works, the installer-vs-manual difference is execution context: elevation (admin installer resolves `%LOCALAPPDATA%` to a different profile) or a different target folder. Escape hatch: a working manual `setup.bat` IS a complete install — the installer only adds git-install + clone + config.key copy, and setup.bat already copies config.key itself.
