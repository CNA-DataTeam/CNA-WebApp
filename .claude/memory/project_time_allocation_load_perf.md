The Time Allocation page (`pages/time-allocation-tool.py`) had a slow INITIAL load but was snappy afterward — the classic "cold cache + UNC share" signature, since this is a locally-run app reading parquet over `\\therestaurantstore.com\...` and each user pays a cold cache on every app launch. The matching/autofill itself was never the bottleneck (it's an O(1) dict lookup on a prebuilt structure). Changes 1–4 (2026-06) cut the cold cost while keeping the warm snappiness; change 5 sped up the admin Exports/Edit-Entries tables:

1. **`_build_account_lookup` was vectorized** (was `df.iterrows()` over ~15k account rows on every cold build — ~450ms; now ~70ms, ~6.5×). Uses per-cell `.map(lambda x: str(x or "").strip())` then groupby/`drop_duplicates`. **Gotcha:** it deliberately keeps the `str(x or "").strip()` per-cell map rather than `astype(str)` because pandas versions disagree on how `astype(str)` renders `NaN` (this app's pandas 2.x preserves `NaN`; older versions emit `"nan"`). The map guarantees byte-identical output to the old loop for any input. Proven equivalent via a 4000-case differential test (since deleted). Preserve the tiebreaks if you touch it: every non-blank Reporting Name appears even with no codes; a Customer Code maps to the Reporting Name of its FIRST row occurrence among rows where both are present.

2. **Channel usage-frequency ordering was removed** — it scanned *every* saved time-allocation parquet across all users/days (a `pyarrow.dataset` scan) on each cold load just to reorder a dropdown. Replaced with a fixed `CHANNEL_OPTIONS` order. Details + the removed symbols are in [[gotchas_time_allocation_editing]] point 2.

3. **Admin tabs now lazy-load.** `st.tabs` normally executes ALL tab bodies every run (tabs are client-side CSS show/hide), so an admin paid the Exports + Admin-Settings **full-dataset reads** (`load_time_allocation_exports` = rglob + read of the whole tree) even while sitting on the Input tab. Fix: `st.tabs([...], on_change="rerun")` makes the tabs stateful so each returned `TabContainer` exposes `.open`; gate with `if input_tab.open: with input_tab: render_...`. **Requires Streamlit ≥1.40-ish for `.open`; confirmed on 1.56.** `.open` is `None` (falsy!) unless `on_change` is `"rerun"` or a callable — if you ever drop `on_change`, every `if tab.open:` becomes false and NOTHING renders. First tab (Input) is open by default on first load. Non-admins skip tabs entirely.

4. **`utils.load_account_lookup` projects columns** — reads only `CustomerCode` / `Reporting Name` / `Company Group USE` (via `pq.read_schema` + `pd.read_parquet(columns=...)`) instead of every column of the wide accounts parquet, with a full-read fallback.

5. **`load_time_allocation_exports` got a `pyarrow.dataset` fast path** (the Exports tab + admin Edit Entries tables). Storage is one tiny parquet per user per day, so the old per-file `pd.read_parquet` loop was latency-bound over UNC (~15s for 658 rows / 175 files here). `_read_exports_via_dataset` reads them in ONE scan with `fragment_readahead=48` (reads ~48 files concurrently, hiding per-file latency) → ~4–5× faster, byte-identical output, with try/except fallback to the per-file reader. **Two load-bearing gotchas:** (a) it passes an **explicit base schema** so a file missing a base column (e.g. legacy file without `Customer Code`) null-fills instead of the column being silently dropped, and so schema inference doesn't depend on the first-discovered file; (b) it builds the dataset from the **explicit `*.parquet` file list** (`_iter_time_allocation_files`), NOT `ds.dataset(directory)` — a directory scan opens *every* file as parquet, so one stray `desktop.ini`/`Thumbs.db` (synced shares create these) or a leftover `*.parquet.tmp` from an interrupted `atomic_write_parquet` makes the whole scan raise and silently defeats the fast path on every load. Per-row `Source File` provenance comes from pyarrow's `__filename` augmented field, converted to the same base-relative path the old loader produced — the **Edit Entries save indexes back into each file by `Source File` + `_source_pos`**, so that string and the within-file row order must match exactly (a stable sort by `Source File` desc preserves both). NOTE: this only makes the read *faster*, not *bounded* — it still reads all history. Windowing the load by date (deferred) is the lever to make it constant regardless of total entries.

6. **Admin Exports + Edit Entries tables: top-100 default behind a "Load All Data" button (2026-06).** Opening either table loads only the ~100 most-recent rows via `_load_recent_exports` (`dataset.head(100)` over the newest-first file list → STOPS reading early, so it's ~constant-time regardless of total history). In this *collapsed* state `_export_filter_panel` shows ONLY the caption "Showing the 100 most-recent entries only. To view all entries and use filters, click **Load All Data**." plus a green **Load All Data** button — the filter form is NOT rendered yet. Clicking the button calls `_enter_full_export_load`, which sets the session `{prefix}_applied` to the **full data date range** (`_export_data_date_bounds` — min/max parsed from filenames only, no parquet reads) and pre-fills `{prefix}_from`/`{prefix}_to` to that range; the next run *expands* the table: it loads the whole window (`_load_exports_window` + a `st.progress` bar) and renders the filter form. **Apply Filters** then re-windows/refines as before (in-memory `_apply_export_filters` on the loaded set; dropdown options come from the loaded data, which is now the full history). `applied is None` is the collapsed-vs-expanded flag; nothing resets it back to None except clearing `{prefix}_applied`. The "Load All Data" and "Apply Filters" buttons are `type="primary"`; their white-on-green label comes from the global CSS `[data-testid^="stBaseButton-primary"]` prefix selector (the `^=` is what also catches form-submit primaries like Apply Filters / the stocking-agreement Generate buttons — exact-match `stBaseButton-primary` does NOT). **Editor gotcha (load-bearing, caused a verified data-loss bug):** the `st.data_editor` key MUST include a content signature of the displayed rows' `(Source File, _source_pos)` identities (`_editor_seed_signature`), NOT just `len(filtered)` — a delete-save in recent mode backfills to the same 100-row count, so a len-only key stays constant and Streamlit reapplies the stale positional Delete diff onto a DIFFERENT on-disk row (silent wrong-row deletion). The signature is stable across autofill reruns (same data) so in-progress edits survive. Every save/delete MUST call `_invalidate_admin_export_tables()` (clears the recent cache AND pops the per-session window frames `{prefix}_loaded_df`/`_loaded_sig` for both tables) or the tables show stale rows. Filter dropdown options come from the LOADED data, so in recent mode they only list values from the recent 100 — the date range (full bounds from fiscal config) is the primary "load more" control; widen it + Apply to surface other users/periods. Save path re-reads each file by `Source File` and is unaffected by windowing because whole files are loaded (the single `head()` boundary file is an in-order prefix, so positions stay correct).

7. **Input-tab save speed (2026-06).** Every save ends in a full `st.rerun()` that re-executes the module body, which calls `get_user_department()` and `is_current_user_admin()`. **Gotcha (cost me a wrong diagnosis): `utils.load_users_table` is ALREADY cached (`@st.cache_data ttl=3600`) — don't "add" caching to it.** The real per-rerun UNC read was `get_user_department`'s **fallback**: it did an UNCACHED direct `pd.read_parquet(users.parquet)` on EVERY call whenever the resolved department was blank — i.e. for any user with a blank/missing Department, it re-read the share on every rerun (including each post-save rerun) even though the cached table already had the answer. Fixed by gating the fallback to `if users_df is None and df.empty:` (only re-read on a genuine empty-load failure; users.parquet is immutable in-session so a fresh read can't resolve a blank dept the cache couldn't). Also `save_records` now passes `skip_path=output_path` to `_replace_user_day_entries` so the per-user/day target file isn't read+deleted before `atomic_write_parquet` overwrites it (the target only holds that user's day, so overwriting is equivalent; legacy root files are still cleaned). **What does NOT reload on a save:** `load_users_table`, `_account_lookup_for_dir` (ttl 1h), and `load_time_allocation_user_window` (ttl 30s + a session "window override" stashing the just-saved rows) are all cached — so account + entry data don't re-read. The remaining save cost is the parquet writes + the full-page re-render, in which the `st_calendar` component re-mounts every render (a Streamlit component cost, not a UNC read).

**Why per-lookup-on-demand was REJECTED** (the idea of looking up each Reporting Name↔Customer Code pair only when chosen, instead of building the whole map up front): it can't help. To populate the two dropdowns you need the complete list of every Reporting Name and Customer Code anyway, which is the full read. Deferring would either re-hit the network per selection (worse, and would kill the post-load snappiness) or still need the prebuilt dicts. The cost is the data load + structure build, not the matching.

**If initial load is still slow after this:** the next suspects are `load_time_allocation_user_window` (per-user window read, cached 30s) and the `load_account_lookup` parquet read latency itself. Add timing logs around the four cold calls before guessing — UNC latency/file-count, not CPU, dominates here.
